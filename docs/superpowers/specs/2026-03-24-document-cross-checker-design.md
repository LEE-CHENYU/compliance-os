# Document Cross-Checker: Design Spec

## Product Summary

A two-track document cross-checker for immigrants that surfaces compliance risks by comparing fields across documents and applying deterministic rules gated on the user's specific situation.

**Track A — Young Professionals (F-1 / OPT / STEM OPT)**
Upload I-983 + employment letter → extract fields → cross-check → targeted follow-up → timeline-based case snapshot → tax upsell.

**Track B — Entrepreneurs (Foreign-owned LLC / C-Corp)**
Answer 3-4 entity questions + upload tax return → extract fields → rule-check → targeted follow-up → entity health snapshot → deeper checks upsell.

**Core principle:** Break down cognitive burden buried in compliance filings into something manageable. Every screen earns the next. No information overload.

---

## Design Decisions (from brainstorming)

| Decision | Choice | Rationale |
|---|---|---|
| Multi-track vs single lane | Single lane per track, no mixing | Each track has its own documents, rules, and mental model |
| Extraction approach | LLM-assisted (OpenAI structured output) | I-983/employment letters vary enough that regex is brittle; LLM is faster to build and more robust |
| Follow-up question engine | Hardcoded rules + LLM phrasing | Deterministic core decides *which* questions; LLM makes them natural with context-aware chip options |
| Result screen frame | Timeline-centered (Track A), Entity-centered (Track B) | F-1 risk is temporal; entity risk is structural |
| Formation doc upload (Track B) | Dropped — replaced by 3-4 questions | Formation docs don't surface severe risks; entity type/state/residency are faster to ask |
| Advisory flags | One line per item: form + question + consequence | Product reduces cognitive burden — no paragraphs, no tag soup |
| Advisory gating | Rule-engine gated on user's situation | Only show what applies (e.g., no FBAR for F-1 students under 5 years in US) |

## User Flow

### Shared Entry (Screen 1)

Landing page with two cards:

- **"I'm on OPT / STEM OPT"** → Track A
- **"I own a US business"** → Track B

Headline: "Check your documents before USCIS does"
Subtext: "We cross-check your immigration and business filings to find mismatches that could trigger scrutiny, penalties, or status loss."

No account required. No login wall. Check ID serves as the bookmark URL — users can return to `/check/{id}` to see results. Security note: check IDs are UUIDs (unguessable), but there is no authentication; anyone with the URL can view results. This is acceptable for MVP.

Privacy note: Documents are uploaded to the server for processing and sent to OpenAI for extraction. They are not shared with any other party. The server deletes uploaded files after extraction is complete.

### Track A: Young Professionals

**A1 — Stage select**
One question: "What's your current stage?"
Options: On STEM OPT / On post-completion OPT / Applying for STEM extension / Pre-completion (CPT) / Not sure

Also ask: "How many years have you been in the US on F-1?" (used to gate advisories like FBAR/FATCA)

**A2 — Upload two documents**
Two labeled drop zones:
- Form I-983 (Training Plan for STEM OPT Students)
- Employment Letter (offer letter, verification letter, or contract)

Privacy note: "Documents sent to OpenAI for field extraction only. Never stored or shared beyond that."

**A3 — Side-by-side extraction**
Show extracted fields from both documents in a comparison grid:

| Field | I-983 | Employment Letter | Status |
|---|---|---|---|
| Job title | Data Analyst | Business Ops Associate | ⚠️ Mismatch |
| Salary | $85,000 | $85,000 | ✓ Match |
| Start date | Jul 1, 2025 | Jul 1, 2025 | ✓ Match |
| Employer name | Acme Corp | Acme Corp | ✓ Match |
| Work location | NYC HQ | Remote — TX | ⚡ Needs review |
| Job duties | (expandable) | (expandable) | (semantic check) |
| Supervisor | Jane Smith | Jane Smith | ✓ Match |
| FT status | Full-time | Full-time | ✓ Match |

Bottom prompt: "N fields need attention. Answer a few questions to complete the check."

**A4 — Targeted follow-up**
Only shown for fields with mismatches or ambiguity. Each question:
- States the specific mismatch found
- Explains why it matters (one sentence)
- Offers chip-based quick replies

Questions are selected by hardcoded rules based on which comparisons failed. Phrasing and chip options are generated by LLM with the mismatch context.

Example rules:
- `job_title_mismatch` → ask which is correct, explain RFE risk
- `location_mismatch` → ask about remote/hybrid, explain DHS site visit
- `duties_low_similarity` → ask if daily work uses degree skills, explain major-relatedness requirement

**A5 — Case snapshot (timeline)**
Timeline showing:
- STEM OPT start date (from I-983)
- Today marker
- Key upcoming dates (12-month evaluation due, STEM OPT end, grace period end)
- Risk markers on the timeline

Findings section:
- "Needs attention" — specific issues with one-line action items
- "Looks good" — matched fields listed compactly

Advisory section ("Also worth checking"):
Each advisory is one line: **Form name** — plain-English question — consequence in 2-4 words.

Advisories are gated by the rule engine based on the user's answers and extracted data. Rules:

| Advisory | Condition to show |
|---|---|
| FBAR (FinCEN 114) | F-1 + >5 years in US, OR transitioned to H-1B |
| Form 8938 (FATCA) | Same as FBAR (requires meeting Substantial Presence Test) |
| AR-11 address change | Location mismatch detected between I-983 and employment letter |
| Multi-state tax | I-983 state ≠ employment letter state/location |
| Investment income reporting | Future: gated on tax upsell findings |

**A6 — Tax upsell**
"Also check your tax filing?"
Lists the 4 specific checks:
- 1040 filed instead of 1040-NR
- Schedule C self-employment on OPT
- Missing Form 3520 for foreign gifts >$100K
- Missing Form 5472 for foreign-owned LLC

Upload slot for most recent tax return (1040, 1040-NR, or 8843).
Options: "Check my taxes too" / "Skip for now"

One labeled drop zone:
- Most recent tax return (1120, 1065, 5472 + pro forma 1120, or Schedule C)

Same pattern as Track A — only ask about detected issues.

Example rules:

**B5 — Entity health snapshot**

|---|---|

| Registered agent | Always for entities using a registered agent service |

| Form 3520 | Foreign capital transfers >$100K |
| FinCEN BOI | Foreign entity registered in US (not US-formed) |

**B6 — Deeper checks upsell**
"Deepen your check?" with cards:
- State compliance check (annual report, good standing)
- FBAR check (2 questions)
- Personal tax cross-check (upload personal return)

## Rule Engine Design

The compliance rule engine is the deterministic core of the product. It gates all findings and advisories — nothing is shown without a rule that says it should be.

### Rule Structure

Each rule is a self-contained unit:

```
Rule:
  id: string                    # e.g. "nra_scorp_invalid"
  track: "stem_opt" | "entity" | "shared"
  type: "comparison" | "logic" | "advisory"
  conditions: Condition[]       # ALL must be true
  severity: "critical" | "warning" | "info"
  finding:
    title: string               # One-line finding
    action: string              # One-line recommended action
    consequence: string         # 2-4 word consequence label
    immigration_impact: bool    # Does this affect immigration status?
```

### Condition Types

```
Condition:
  field: string                 # e.g. "stage", "years_in_us", "entity_type"
  operator: "eq" | "neq" | "gt" | "lt" | "in" | "contains" | "missing" | "mismatch"
  value: any                    # comparison target
  source: "answers" | "extraction_a" | "extraction_b" | "comparison"
```

**Operator semantics:**

- `mismatch` with `source: comparison` — resolves to `comparisons[field].status in ("mismatch", "needs_review")`. A field matches if its comparison status is `"match"`.
- `missing` with `source: extraction_a` or `extraction_b` — true if the field was not extracted or has null/empty value.
- `gt` / `lt` on date fields — compares extracted date against a computed date value.

**Date magic values:**

Rules can reference computed dates in the `value` field:
- `"today"` — current date at evaluation time
- `"N_months_ago"` (e.g., `"12_months_ago"`) — today minus N months
- `"N_days_from_now"` (e.g., `"60_days_from_now"`) — today plus N days

For date comparisons: `field: start_date, operator: lt, value: "12_months_ago"` means "the start date is earlier than 12 months ago" (i.e., more than 12 months have passed since start).

### Rule Engine Implementation

```
RuleEngine:
  rules: Rule[]

  evaluate(context: EvaluationContext) -> Finding[]:
    findings = []
    for rule in rules:
      if all(condition.evaluate(context) for condition in rule.conditions):
        findings.append(rule.finding)
    return sort_by_severity(findings)

EvaluationContext:
  answers: dict          # User's answers from stage select / entity info
  extraction_a: dict     # Fields from document A (I-983 or answers)
  extraction_b: dict     # Fields from document B (employment letter or tax return)
  comparisons: dict      # Results of field-by-field comparison
  today: date            # Current date for timeline calculations
```

Rules are stored as YAML config files, not hardcoded in application code. This allows:
- Adding new rules without code changes
- Versioning rule sets
- Testing rules independently
- Future: per-jurisdiction rule variants

### Two-Pass Evaluation

The rule engine runs twice:

1. **First pass** — after extraction + comparison, using initial answers. Produces findings and determines which follow-up questions to ask.
2. **Second pass** — after follow-up answers are submitted, answers dict is updated with follow-up responses, and the engine re-evaluates. This allows advisories gated on follow-up answers (e.g., `advisory_form_3520` depends on `foreign_capital_transfer` answer from follow-up or intake) to fire correctly.

The final snapshot shows findings from the second pass only.

### Rule Versioning

Each YAML rule file has a `version` field (semver). The `findings` table stores `rule_version` so historical findings remain stable even if rule text is later updated.

### Field Mapping (I-983 → Employment Letter)

The two documents use different names for equivalent fields. The comparison engine uses this mapping:

| Comparison field | I-983 schema field | Employment letter schema field | Match type |
|---|---|---|---|
| `job_title` | `job_title` | `job_title` | fuzzy |
| `employer_name` | `employer_name` | `employer_name` | exact |
| `work_location` | `work_site_address` | `work_location` | fuzzy |
| `start_date` | `start_date` | `start_date` | exact |
| `compensation` | `compensation` | `compensation` | numeric |
| `duties` | `duties_description` | `duties_description` | semantic |
| `supervisor` | `supervisor_name` | `manager_name` | fuzzy |
| `full_time` | `full_time` | `full_time` | exact |

### LLM Extraction Strategy

Each document type gets a separate extraction prompt with the corresponding schema as structured output format.

- **I-983:** Government form with semi-structured layout. Send full text. Prompt includes field descriptions and expected formats (e.g., "SEVIS number: N followed by 10 digits").
- **Employment letter:** Freeform business letter. Send full text. Prompt includes field descriptions and notes that some fields may not be present.
- **Tax return:** Structured government form. Send full text. Prompt focuses on identifying form type, schedules present, and key amounts.

All extractions use OpenAI's structured output mode (JSON schema response format) to ensure typed, parseable results.

### Extraction Failure Handling

If extraction fails or returns low-confidence results:

- **Complete failure** (wrong file type, corrupted PDF, password-protected): Show error message "We couldn't read this file. Please upload a PDF of your [document type]." Allow re-upload.
- **Partial extraction** (some fields extracted, others missing): Show extracted fields with empty cells for missing ones. Mark missing fields as "Could not extract — you can fill this in manually." Proceed with comparison on available fields only.
- **Low confidence** (confidence < 0.5 on a field): Show the value with a "Verify" tag. User can confirm or correct. Low-confidence fields are excluded from auto-comparison but included if user confirms.

Confidence threshold for auto-comparison: **0.7**. Below this, the field is flagged for manual review.

## Comparison Engine

### Match Types

| Type | Method | Used for |
|---|---|---|
| **Exact** | String equality (case-insensitive, whitespace-normalized) | EIN, start date, employer name |
| **Fuzzy** | Levenshtein distance or token overlap, threshold 0.85 | Job title, location |
| **Numeric** | Absolute difference within tolerance (±$500 or ±2%) | Compensation |
| **Semantic** | LLM similarity score 0-1 | Job duties vs training goals, duties vs major |
| **Logic** | Rule-based evaluation | NRA + S-Corp, missing 5472, SPT check |

### Comparison Output

Each comparison produces:

```yaml
comparison:
  field_name: string
  value_a: string           # From document A
  value_b: string           # From document B
  match_type: string        # exact, fuzzy, numeric, semantic
  status: string            # match, mismatch, needs_review
  confidence: float         # 0-1
  detail: string | null     # For semantic: explanation of similarity/divergence
```

```sql
checks (
  id            UUID PRIMARY KEY,
  track         TEXT NOT NULL,          -- 'stem_opt' | 'entity'
  stage         TEXT,                   -- 'stem_opt' | 'opt' | 'applying_stem' | 'pre_completion'
  status        TEXT DEFAULT 'intake',  -- 'intake' | 'uploaded' | 'extracted' | 'reviewed' | 'saved'
  answers       JSON DEFAULT '{}',     -- All user answers (stage, years_in_us, entity_type, etc.)
  created_at    TIMESTAMP,
  updated_at    TIMESTAMP
)

-- Uploaded documents
documents (
  id            UUID PRIMARY KEY,
  check_id      UUID REFERENCES checks,
  doc_type      TEXT NOT NULL,          -- 'i983' | 'employment_letter' | 'tax_return'
  filename      TEXT,
  file_path     TEXT,
  file_size     INTEGER,
  mime_type     TEXT,
  uploaded_at   TIMESTAMP
)

-- Extracted fields per document
extracted_fields (
  id            UUID PRIMARY KEY,
  document_id   UUID REFERENCES documents,
  field_name    TEXT NOT NULL,
  field_value   TEXT,
  confidence    REAL,
  raw_text      TEXT                    -- Original text the value was extracted from
)

-- Field comparisons between two documents
comparisons (
  id            UUID PRIMARY KEY,
  check_id      UUID REFERENCES checks,
  field_name    TEXT NOT NULL,
  value_a       TEXT,
  value_b       TEXT,
  match_type    TEXT,                   -- 'exact' | 'fuzzy' | 'numeric' | 'semantic'
  status        TEXT,                   -- 'match' | 'mismatch' | 'needs_review'
  confidence    REAL,
  detail        TEXT
)

-- Follow-up questions and answers
followups (
  id            UUID PRIMARY KEY,
  check_id      UUID REFERENCES checks,
  question_key  TEXT NOT NULL,          -- Maps to rule ID that triggered it
  question_text TEXT,
  chips         JSON,                   -- Available quick-reply options
  answer        TEXT,
  answered_at   TIMESTAMP
)

-- Final findings (from rule engine)
findings (
  id            UUID PRIMARY KEY,
  check_id      UUID REFERENCES checks,
  rule_id       TEXT NOT NULL,          -- Which rule produced this
  rule_version  TEXT,                   -- Semver of rule file at evaluation time
  severity      TEXT,                   -- 'critical' | 'warning' | 'info'
  category      TEXT,                   -- 'comparison' | 'logic' | 'advisory'
  title         TEXT,
  action        TEXT,
  consequence   TEXT,
  immigration_impact BOOLEAN,
  source_comparison_id UUID REFERENCES comparisons  -- NULL for logic/advisory rules
)
```

## API Endpoints

```
POST   /api/checks                          Create a new check session
GET    /api/checks/{id}                     Get check status and results
PATCH  /api/checks/{id}                     Update answers

POST   /api/checks/{id}/documents           Upload a document
GET    /api/checks/{id}/documents           List uploaded documents

POST   /api/checks/{id}/extract             Run LLM extraction on uploaded documents
GET    /api/checks/{id}/extractions         Get extracted fields

POST   /api/checks/{id}/compare             Run field comparisons
GET    /api/checks/{id}/comparisons         Get comparison results

POST   /api/checks/{id}/evaluate            Run rule engine on comparisons + answers
GET    /api/checks/{id}/findings            Get all findings (sorted by severity)

POST   /api/checks/{id}/followups           Generate follow-up questions
PATCH  /api/checks/{id}/followups/{fid}     Submit answer to follow-up
GET    /api/checks/{id}/followups           Get follow-up questions and answers

POST   /api/checks/{id}/snapshot            Generate final case snapshot
POST   /api/checks/{id}/save                Convert to persistent case workspace
```

Each `/review` route is a single-page flow that progresses through extraction preview → follow-up questions → case snapshot without page reloads. Progress is shown inline.

## What Gets Removed

The current codebase has a multi-track wizard (14 steps), data room, chat panel, and checklist system. These are replaced by the simpler check flow:

| Old | New |
|---|---|
| 14-step wizard with conditional tracks | 1-2 questions per track |
| Document data room with checklist slots | 2 labeled upload zones (Track A) or 1 (Track B) |
| Chat panel with follow-up Q&A | Inline follow-up with chips (no chat UI) |
| `cases` table | `checks` table |
| `discovery_answers` table | `checks.answers` JSON field |
| `chat_messages` table | `followups` table |
| `documents` table | `documents` table (simplified) |
| No extraction | `extracted_fields` table + LLM extraction |
| No comparison | `comparisons` table + comparison engine |
| No rule engine | `findings` table + YAML rule engine |

Git history preserves all old code. The wizard can be brought back later if needed for a deeper "workspace" mode.
